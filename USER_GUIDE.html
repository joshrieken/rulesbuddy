Rule Maven — Developer & Architect Guide

A RAG-based board-game rules Q&A app. Upload a rulebook, ask plain-English questions, get answers that quote the exact page. This guide is for the people building and operating it — the why behind the structure, not just the what.

There are three guides in this repo. This one is for developers and architects. ADMIN_GUIDE.html is a task guide for admins, and USERS_GUIDE.html is for regular users. The latter two are prototypes destined for an in-app help section.

What it is

Rule Maven turns tabletop rulebooks into a searchable knowledge base. A user builds a library of games, attaches each game's rulebook sources (PDF, Word, HTML, spreadsheet, image, or pasted text), and asks questions like "Can I build on someone else's road?". The app answers with retrieval-augmented generation: embed the question, retrieve the most relevant rulebook chunks, have an LLM answer only from those chunks, and cite the page it used.

The defining product constraints — and the reasons most of the architecture exists — are: (1) accuracy over cost (a wrong rules answer is worse than no answer), (2) copyright safety (never expose the source text), and (3) responsiveness (nothing slow ever blocks the UI).

Architecture at a glance

Standard Phoenix context layering, with two cross-cutting subsystems (extraction and answering) that are almost entirely asynchronous.

LiveViews (*_live)→ Context modules (Games, Faq, Users…)→ Schemas + Repo (Postgres/pgvector)
LiveView enqueues→ Oban worker does the slow work→ PubSub broadcast→ LiveView updates live

The golden rule, enforced throughout: the LiveView process never blocks on slow work. Fire-and-forget work goes to Oban; slow interactive work uses start_async; results stream back over Phoenix.PubSub. A user navigating away, or a server restart, never loses an in-flight extraction or answer.

Data model

The core entities and how they relate:

EntityRole
GameA library entry: name, category, BGG metadata, generated suggestions/cheat sheet. Self-referencing via parent_game_id for expansions.
ExpansionNot a separate table — a Game with a parent_game_id. Its rules can be opted into the answer scope of the base game.
Document (Source)One rulebook attached to a game. Stored page-by-page; each page carries raw text, optional cleaned text, a printed page number, and an extraction-confidence score. A published flag gates retrievability.
ChunkA small slice of a page's effective text (cleaned || text), embedded into a 768-dim vector. The unit retrieval searches over.
QuestionLogOne asked question and its answer, plus cited_passage/cited_page, verdict, embeddings, pooling/trust fields, threading (parent_question_id), and visibility.
QuestionVoteA community up/down vote ("up"/"down") on a QuestionLog, weighted by voter reputation. Unique per (question, user).
UserAccount with a role and an earned reputation that weights their votes.

Effective text rule: anywhere the app uses page text — chunking, embedding, citations — it reads cleaned || text. Cleaning is optional and reversible; it never destroys the raw extraction.

Capability-based authorization

Authorization is gated by capability, never role name. Roles map to capability lists; all checks flow through one function.

# lib/rule_maven/users/user.ex
@all_roles ["user", "admin"]
@role_capabilities %{
  "user"  => [],
  "admin" => [:admin]
}
def can?(%{role: role}, capability),
  do: capability in Map.get(@role_capabilities, role, [])

Call sites use RuleMaven.Users.can?(user, :admin) — never a role == … string comparison. To add a role, extend @all_roles and give it a capability list; to grant or revoke a power, edit a capability list. Nothing else changes.

Why this matters architecturally: new roles (e.g. a "moderator" with a subset of admin powers) drop in by adding capabilities like :moderate_questions and gating the relevant call sites on those, without touching role-name conditionals scattered through the codebase.

Extraction pipeline — accuracy first

Turning a rulebook into clean, page-attributed text is the hardest part of the system, and the most cost-sensitive. The strategy: spend the expensive model only where it measurably improves accuracy.

  • Native-text formats (Word, HTML, spreadsheet, plain text/Markdown) carry their text exactly — read structurally, no OCR, no AI. Cheapest path to a perfect copy.
  • PDFs with a clean text layer are trusted as-is, for free.
  • Dirty PDF pages get read two cheap, independent ways — the PDF text or local OCR vs. a fast AI vision read — and the two transcriptions are compared. Agreement ⇒ accept.
  • Only disagreeing pages escalate to a stronger, higher-resolution vision model with an adversarial critic: a second AI hunts the transcription for missing text, hallucinations, wrong numbers, and dropped table rows; the page is corrected and re-checked.

Scanned PDFs are rasterized at 300 DPI grayscale and read in parallel across cores. Page detection recovers printed page numbers — even when numbering shifts mid-book — and extrapolates front/back matter so reader navigation and citations line up with the physical book.

Choosing the cheap vision reader is decided by its escalation rate, not its raw transcription quality — a reader that triggers fewer expensive escalations is cheaper overall even if slightly noisier. There's an A/B harness (mix vision_ab) for this.

Every page keeps a confidence from the cross-check. Low-confidence pages are surfaced for human review in the reader, where an admin can hand-edit or trigger a stronger re-extract (durable ReextractPageWorker). Optional cleanup (CleanupWorker) LLM-rewrites messy pages a level at a time, resumably, writing each page back as it finishes.

Retrieval & answering

An ask flows through LLM.ask/… → (pool check) → call_llm/… → AskWorker persistence + broadcast.

Embed question (768-dim)→ Pool hit? serve cached→ else retrieve top chunks→ build system prompt from chunk context→ LLM answers + cites page→ recover page if dropped→ maybe pool

Retrieval ranks chunks by cosine similarity over published, embedded documents, pulls in cross-referenced chunks, and falls back to full-text then keyword search if vectors come up short. Only documents that pass a quality check are published and therefore retrievable; lower-quality uploads sit in pending review.

Page-citation is a hard requirement. The worker trusts the model's [Page N] marker first, else parses it from the cited passage, else recovers it by matching the passage back to its source chunk (each chunk text carries a [Page N] prefix). The displayed passage has those markers stripped after the page is extracted.

Answer pool & staleness

Before any retrieval or LLM call, the app checks whether a very similar question was already answered well. If a pooled/community answer is close enough (cosine ≥ 0.92 by default, trusted answers first), it serves that answer instantly — no retrieval, no LLM, no cost.

Pool hits carry a tier: :trusted (community-verified) or :provisional (a single-source reuse not yet community-confirmed). The tier is encoded in the served row's model field ("cached" vs "cached-unverified") so it survives a page reload. A served pool hit carries the source row's cited_passage/cited_page and source_question_log_id — so reuse keeps the original citation and votes accrue back to the canonical row.

Staleness invariant: when rulebook content changes (new doc, edited text, cleanup, deletion), the pool for that game is invalidated. Auto-pooled answers are demoted; community answers are flagged needs_review (skipped by pool lookup) until an admin re-approves. Stale cached answers must never outlive the text they cited.

Trust & confidence model

Two distinct signals, often confused — keep them separate:

Trust score (ranking / promotion)

Games.Trust computes a per-question trust_score from reputation-weighted net votes + a citation bonus, with pinned as an override. A user's reputation (earned from the net votes their authored answers receive) sets their vote weight: min(1 + ln(1 + rep), cap), so a brand-new user weighs ~1.0 and trusted contributors weigh more, up to a cap. Private favorites are deliberately excluded from trust.

Confidence meter (per-answer UI signal)

A pure heuristic over existing signals — there is no stored confidence column. It maps each answer to one of five levels shown as a segmented meter:

LevelLabelWhen
5 · VerifiedCommunity-verifiedTrusted (non-provisional) pool hit.
4 · StrongCited from rulebookHas both a cited passage and a page.
3 · GoodCited passage, page unconfirmedHas a passage but no confirmed page.
2 · FairUnverified — single sourceProvisional pool hit with no citation.
1 · LowNo direct citationModel answered without a matched passage.
Design subtlety: the citation checks are ordered above the provisional-pool check. A reused answer carries the source's citation, so it must read at its citation strength — the same as a fresh ask — rather than dropping a level just because it came from the pool. Only a trusted pool hit short-circuits to Verified. (This fixed a "Strong → Fair on reuse" regression.) A help bubble next to the meter explains the level and the path to the next one.

Copyright lockdown

Hard rule. Rulebooks may be copyrighted, so the original is never exposed. Original PDFs are not served over HTTP at all — kept on disk only for extraction. The extracted-text "View as HTML" view is admin-only; for non-admins the URL 404s, so the file's existence isn't even revealed. Regular users see only the source name in a game's rulebook list. The only rulebook text any user sees is the cited passage inside an answer.

Running & config

mix setup            # install deps, set up DB
mix phx.server       # start the server
iex -S mix phx.server  # with a console

Open localhost:4000. Server logs land in tmp/phoenix.log / var/log/phoenix.log. You need: PostgreSQL with pgvector, and API keys for your chosen LLM, vision, and embedding providers. Provider/model selection lives in /settings.

The embedding model's dimensionality is pinned to 768 to match the search index. Changing it corrupts existing vectors — only do it as part of a full re-embed.

Extending the app

  • New role/permission → add to @all_roles + @role_capabilities, gate call sites on can?/2. Never branch on a role name.
  • New slow feature → write an Oban worker, broadcast over PubSub, subscribe in the LiveView. Never block the socket. Convert any fire-and-forget to a job; use start_async for slow interactive work.
  • New AI provider → implement against the LLM/Embed seams; keep embedding output at 768 dims.
  • New extraction format → if it carries exact text, read it structurally (no AI); only fall back to the vision cross-check for image-like input.
  • Touching cached answers → if you change rulebook content, you must invalidate the pool. Stale citations are a correctness bug, not a cosmetic one.
