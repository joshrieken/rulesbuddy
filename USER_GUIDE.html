Rule Maven — User Guide

Upload a board-game rulebook, then ask plain-English rules questions and get cited answers pulled straight from the book.

What it is

Rule Maven is a Phoenix LiveView web app for tabletop gamers. You build a library of games, attach each game's rulebook(s) (PDF or pasted text), and the app turns them into a searchable knowledge base. Ask a question like "Can I build on someone else's road?" and it answers using retrieval-augmented generation (RAG) — finding the relevant rulebook passages and quoting the page they came from.

It supports board games, card games, tabletop RPGs, TCGs, video games, and more — including expansions that hang off a parent game.

Cited answers
Every answer points back to the rulebook page it used.
Community pool
Good answers get reused, so repeat questions are instant and free.

Accounts & roles

Access is gated by login. You register an account (an invite code may be required), sign in at /login, and land on the game library.

RoleCan do

admin / game masterEverything a user can, plus: view a rulebook's extracted-text HTML, and the /admin dashboard — review rulebooks, moderate questions & threads, manage users, invites, the game catalog, security, and the database/Oban dashboards.
Copyright lockdown. Rulebooks may be copyrighted, so end users can never download the original PDF or read the full text. Original PDFs are not served over HTTP at all (kept on disk only for extraction). The extracted-text "View as HTML" view is admin-only — for non-admins the URL 404s so the file's existence isn't even revealed. Regular users see only the source name in a game's rulebook list. Answers still quote the cited page; that's the only rulebook text users see.

The game library

The home page (/) is your library. It's built for fast browsing:

  • Search — type to filter by name; results update live.
  • Category filter — narrow to board games, RPGs, TCGs, etc.
  • Views — switch how the list is displayed.
  • Expansions — toggle a game open to see its expansions inline.
  • Collection & favorites — mark games as owned (your collection) or favorite them for quick access.
  • Keyboard nav — move through the list and open a game without the mouse.
  • BGG actions — pull metadata (pull_bgg) or fetch a game's expansions (pull_expansions) on demand.
  • Request support — flag a game you want the catalog to cover.

Click any game to open its Q&A page.

Adding a game

Use New Game (/games/new) to open the form. The same form edits an existing game (/games/:id/edit).

  1. Enter the name and pick a category (board_game, card_game, tabletop_rpg, tcg, video_game, other).
  2. Optionally link a parent game to make this an expansion.
  3. Let it pull BGG metadata (or refresh later). The app can also auto-suggest categories and related info via the LLM.
  4. Attach one or more rulebook sources (next section).
  5. Save. Embedding and cheat-sheet generation kick off in the background.
Suggestions & categories are generated by background LLM workers. You can regenerate or clear them from the form if they look off.

Rulebooks & cleanup

On the game form you manage sources — the rulebooks behind every answer.

Adding a source

  • Upload a PDF — extraction runs in the background (durable DownloadWorker, "upload" mode), so a scanned PDF's OCR no longer freezes the page. A processing banner with a Cancel button shows on the Manage tab while it works. Batch upload of several PDFs at once is supported.
  • Give a URL — the same background worker fetches and extracts the PDF (timeouts, PDF validation, size caps, and a Cancel button guard the download).
  • Paste text — for rulebooks you already have as text.

Scanned PDFs are OCR'd automatically (rasterized at 200 DPI, OCR'd in parallel across cores). Page detection finds printed page numbers — even when numbering shifts mid-book — and extrapolates front/back matter, so reader page-nav and citations line up with the real book.

Each source is stored as pages. From the game form you can open a source in the built-in reader (scroll or paginated mode), step through pages, and hand-edit any page's text. Editing text re-chunks, re-embeds, and refreshes generated suggestions/categories for the game.

✨ Clean up text

Raw PDF extraction is messy — broken line wraps, headers/footers, column soup. The cleanup button runs each page through an LLM to produce a clean version. It's labeled Clean on the first run and Wipe & clean once cleaned text already exists; Clean again re-cleans the already-cleaned text for a further pass.

Pick a strength level — Light, Standard, or Aggressive — to control how much the LLM rewrites. The choice is remembered per browser. Cleanup also strips the printed page number from the body (it's stored separately).

How it works: A durable background job (CleanupWorker) cleans pages one at a time and writes each cleaned page back as it finishes — you watch a live, restart-proof progress counter (cleaning_done). It's resumable: if the server restarts mid-clean, the job picks up the remaining pages. Cleaned text auto-applies and stays editable afterward.
"Wipe & clean" re-cleans all pages from scratch (it clears prior cleaned text first), then re-chunks the document so the clean text feeds search. The extracted-text HTML view is regenerated from the cleaned text too.

What "effective text" means

For every page the app uses cleaned || text — the cleaned version if it exists, otherwise the raw extraction. That "effective" text is what gets chunked, embedded, and quoted in answers.

Asking questions

The game's Show page (/games/:id) is where you ask. Type a natural-language question and submit.

  • Quick-ask & suggestions — one-click common questions to get started.
  • Expansion toggle — include an expansion's rules in the answer scope.
  • Threads — questions are grouped into conversation threads; multi-turn follow-ups keep context. Switch between threads in the sidebar.
  • Search — search within this game's existing questions/answers.
Answers are asynchronous. When you ask, a placeholder appears immediately and a background AskWorker fills it in, then pushes the result to your screen live. You don't have to wait on the page.

FAQ & the community pool

Two things make repeat questions cheap and fast:

Answer pool

Before running the LLM, the app checks whether a very similar question was already answered well. If a pooled or community answer is close enough (cosine similarity ≥ 0.92 by default, trusted answers first), it serves that answer instantly — no retrieval, no LLM call, no cost.

When rulebook content changes (new doc, edited text, cleanup, deletion), the pool for that game is invalidated so stale cached answers stop serving. Auto-pooled answers get demoted; community answers are flagged needs_review (skipped by pool lookup) until an admin re-approves them.

FAQ page

The FAQ page (/games/:id/faq) collects the community's notable Q&A for a game. You can filter by category. Admins can promote answers into the canonical FAQ or reject them.

Cheat sheets

Each game can have an auto-generated cheat sheet — a condensed rules summary. Generation is async (generate_async) at a chosen detail level (e.g. compact), and can include selected expansions.

  • View at /games/:id/cheatsheet; specific saved versions at /games/:id/cheatsheet/:version_id.
  • Multiple versions are kept; set one as active or delete old ones.
  • Generation status, provider, model, level, and timing are tracked while it runs.

Import from BoardGameGeek

The Import page (/games/import) bootstraps your library from BGG:

  1. Enter your BGG username and fetch your collection.
  2. Review the games it found.
  3. Import — they're added to your library with BGG metadata attached.

Settings

/settings manages your profile and AI providers:

  • Profile — update username and email.
  • LLM provider — choose which model provider answers questions.
  • Embedding provider — choose the provider used to vectorize text. (The default embedding model is pinned to 768 dimensions to match the search index.)
  • Usage — /settings/usage shows your AI usage/stats.
Changing the embedding model's dimensions would corrupt existing search vectors — the app pins this deliberately. Leave it unless you know you're re-embedding everything.

Behind the scenes — how an answer is built

From upload to answer:

Upload / download PDF→ Extract pages (OCR if scanned) + detect page numbers→ (optional) Clean text→ Chunk→ Embed (768-dim vectors)
You ask a question→ Embed question→ Pool hit? serve cached→ else retrieve top chunks→ LLM answers + cites page→ Maybe pool the answer

Retrieval ranks chunks by vector (cosine) similarity over published, embedded documents, pulls in cross-referenced chunks, and falls back to full-text then keyword search if vectors come up short. Only documents that pass a quality check are published and therefore retrievable; lower-quality uploads sit in pending review until an admin approves them.

Running the app (developers)

mix setup            # install deps, set up DB
mix phx.server       # start the server
# or, with a console:
iex -S mix phx.server

Then open localhost:4000. Server logs land in tmp/phoenix.log / var/log/phoenix.log.

You'll need API keys for your chosen LLM and embedding providers configured in the environment, plus PostgreSQL with the pgvector extension for vector search.
